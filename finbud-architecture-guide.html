1. MVC là gì và áp dụng vào project này ra sao?

Nào, trước tiên mình giải thích khái niệm cái đã. MVC viết tắt của Model – View – Controller. Đây là cách tổ chức code sao cho mỗi phần chỉ lo đúng một việc thôi — giống như trong nhà hàng vậy: đầu bếp (Model) lo nấu ăn, bồi bàn (Controller) lo nhận order và phục vụ, còn thực khách (View) chỉ việc nhìn thấy món ăn đẹp đẹp và thưởng thức.

Tách ra như vậy thì code dễ đọc hơn, dễ sửa hơn, và khi team có nhiều người thì mỗi người lo một phần mà không đụng vào nhau.

MODEL — Tầng Dữ Liệu

Các bạn cứ hình dung Model như cái kho hàng của nhà hàng vậy. Nó định nghĩa "hình dáng" của dữ liệu (user trông như nào, goal có những trường gì...) và lo việc đọc/ghi vào database.

Mở file backend/models/User.js ra, các bạn sẽ thấy thế này:

const userSchema = new mongoose.Schema({
    email: {
      type: String,
      required: true,       // bắt buộc phải có — thiếu là báo lỗi
      unique: true,          // không cho 2 user trùng email
      trim: true,            // tự cắt khoảng trắng 2 đầu
      lowercase: true,       // tự chuyển thành chữ thường
    },
    accountData: {
      username: { type: String, required: true, unique: true },
      password: String,      // sẽ được mã hoá (hash) trước khi lưu
      priviledge: { type: String, enum: ["admin", "user"], default: "user" },
    },
    identityData: {
      firstName: String,
      lastName: String,
      displayName: String,
      profilePicture: String,
    },
    bankingAccountData: {
      accountBalance: { type: Number, default: 0 },
      cash: { type: Number, default: 1000 },
    },
});
L4mongoose.Schema({...}) — Đây là "khuôn mẫu". Tưởng tượng nó như tờ đơn đăng ký: phải điền đúng các ô đã quy định. Mỗi user trong MongoDB sẽ có đúng các trường như khai báo ở đây.
L8unique: true — MongoDB sẽ tự tạo index và từ chối nếu có 2 user cùng email. Giống như số CMND vậy — không ai được trùng.
L19password: String — Để ý là không có required nhé! Lý do: user đăng nhập bằng Google sẽ không có password. Mình sẽ nói kỹ hơn ở phần OAuth bên dưới.
Thầy lưu ý: Model không chỉ định nghĩa schema đâu, nó còn chứa cả "logic liên quan đến dữ liệu" nữa. Ví dụ: tự động mã hoá password trước khi lưu, hoặc so sánh password khi đăng nhập. 
// Tự động hash password TRƯỚC KHI lưu vào database
userSchema.pre('save', async function() {
  if (this.accountData.password && this.isModified('accountData.password')) {
      const salt = await bcryptjs.genSalt(10);
      this.accountData.password = await bcryptjs.hash(this.accountData.password, salt);
  }
});

// Hàm để so sánh password khi đăng nhập
userSchema.methods.comparePassword = async function (enteredPassword) {
    if (!this.accountData.password) return false; // User OAuth không có password
    return await bcryptjs.compare(enteredPassword, this.accountData.password);
};
L63pre('save') — Cái này gọi là Mongoose hook (móc nối). Mỗi khi gọi .save(), hàm này sẽ chạy trước để mã hoá password. User nhập "abc123" thì DB lưu thành chuỗi hash dài loằng ngoằng — an toàn hơn rất nhiều.
L64isModified('accountData.password') — Chỉ hash lại khi password thực sự thay đổi. Tại sao? Vì nếu không kiểm tra, mỗi lần save user (ví dụ đổi tên), nó lại hash cái password đã hash → sai bét!

CONTROLLER — Tầng Xử Lý Logic

Controller giống như anh bồi bàn: nhận yêu cầu từ khách (request từ frontend), đi vào bếp lấy đồ (gọi Model), rồi bưng ra cho khách (trả response). Trong project này, Controller được tách thành 2 phần nhỏ:

  • Routes (routes/*.js) — Giống bảng menu: "gọi món gì thì đi bàn nào". Nó map URL → hàm xử lý.
  • Controllers (controllers/*.js) — Nơi chứa logic xử lý chính: kiểm tra, truy vấn DB, trả kết quả.

Ví dụ: khi user tạo một Goal (mục tiêu tài chính) mới:

// routes/goalRoutes.js — "Bảng menu"
router.use(isAuthenticated);          // Tất cả route này phải đăng nhập mới dùng được
router.post('/', createGoal);         // POST /api/goals → gọi hàm createGoal

// controllers/goalController.js — "Bếp" xử lý
export const createGoal = async (req, res) => {
    try {
        const goal = await Goal.create({  // Nhờ MODEL tạo dữ liệu mới trong DB
            ...req.body,                  // Lấy data từ body của request
            userId: req.user._id,         // Gắn userId từ session (ai đang đăng nhập)
        });
        res.status(201).json(goal);       // Trả về 201 Created kèm data vừa tạo
    } catch (error) {
        res.status(500).json({ error: 'Failed to create goal' });
    }
};

VIEW — Tầng Giao Diện

View là toàn bộ app React phía frontend. Nhiệm vụ của nó là hiển thị dữ liệu cho user thấy, và gửi request về backend khi user tương tác (bấm nút, submit form,...).

// frontend/src/pages/GoalPage.jsx
const handleCreate = async (e) => {
    e.preventDefault();
    await api.post('/goals', {             // Gửi HTTP POST về backend
        ...form,
        targetAmount: Number(form.targetAmount),
    });
    fetchGoals();                          // Tải lại danh sách goals
};
Tóm lại: Khi user bấm "Tạo Goal" trên giao diện → View gửi request → Controller nhận request, xử lý logic → gọi Model lưu vào database → trả kết quả ngược lại cho View hiển thị. Ba tầng phối hợp rất rõ ràng!

2. Backend kết nối với Frontend bằng cách nào?

Đây là câu hỏi nhiều bạn mới hay thắc mắc. Câu trả lời: Frontend và Backend là hai ứng dụng HOÀN TOÀN TÁCH BIỆT, chạy trên 2 port khác nhau. Chúng "nói chuyện" với nhau thông qua HTTP requests — giống như bạn gửi thư đi rồi nhận thư trả lời vậy.

Để hai bên "nói chuyện" được, cần cấu hình 3 thứ:

Thứ nhất: Frontend tạo "người đưa thư" — Axios instance

frontend/src/services/api.js

import axios from 'axios';

const api = axios.create({
  baseURL: 'http://localhost:5000/api',  // Mọi request đều bắt đầu bằng URL này
  withCredentials: true,                 // Gửi cookie (session) kèm theo mỗi request
});

export default api;
L4baseURL — Thay vì lần nào cũng viết dài dòng axios.get('http://localhost:5000/api/goals'), giờ chỉ cần viết gọn api.get('/goals'). Tiện hơn nhiều!
L5withCredentials: true — Dòng này cực kỳ quan trọng! Nó bảo browser: "Mỗi lần gửi request, nhớ đính kèm cookie session nhé." Nếu thiếu dòng này, backend sẽ không biết bạn là ai, dù bạn đã đăng nhập rồi.

Thứ hai: Backend cho phép Frontend truy cập — cấu hình CORS

backend/server.js

app.use(cors({
  origin: 'http://localhost:5173',  // Chỉ cho phép frontend ở địa chỉ này gọi API
  credentials: true,                // Cho phép gửi/nhận cookie qua lại
}));
Giải thích CORS cho dễ hiểu: Mặc định, browser sẽ chặn mọi request từ trang web này sang trang web khác (vì lý do bảo mật). Frontend ở port 5173, backend ở port 5000 — khác "nguồn gốc" (origin) → bị chặn. Cấu hình CORS giống như backend nói với browser: "Ê, tao cho phép thằng frontend kia gọi API của tao, đừng chặn nó."
L28credentials: true — Phải bật ở CẢ HAI BÊN: frontend (withCredentials) + backend (credentials). Thiếu một bên là cookie không gửi được → session không hoạt động.

Thứ ba: Backend định nghĩa các "đường dẫn" API

// backend/server.js
app.use('/api/auth', authRoutes);    // Mọi request /api/auth/* → authRoutes xử lý
app.use('/api/goals', goalRoutes);   // Mọi request /api/goals/* → goalRoutes xử lý
app.use('/api/threads', threadRoutes);
app.use('/api/chats', chatRoutes);
app.use('/api/notis', notiRoutes);
app.use('/api/stocks', stockRoutes);
app.use('/api/users', userRoutes);
L78app.use('/api/auth', authRoutes) — Khi frontend gọi api.post('/auth/login'), Express sẽ thấy URL bắt đầu bằng /api/auth → chuyển sang authRoutes xử lý. Trong đó, authRoutes thấy phần còn lại là /login → gọi hàm login.

3. Một API request đi qua những bước nào?

OK, bây giờ mình đi vào thực tế luôn nhé. Hãy cùng theo dõi một ví dụ cụ thể: User gửi tin nhắn trong Chat (POST /api/chats/:threadId). Mình sẽ đi từng bước một.

Đi vào code cụ thể nhé:

Bước 1–2: frontend/src/pages/ChatPage.jsx — User bấm nút Gửi

const handleSend = async (e) => {
    e.preventDefault();
    const text = input.trim();
    if (!text || !activeThreadId) return;

    setInput('');                    // Xoá ô nhập liệu ngay lập tức
    setSending(true);                // Hiển thị trạng thái "đang gửi..."
    
    // Thêm 1 tin nhắn "tạm" để user thấy ngay (gọi là optimistic UI)
    setChats((prev) => [...prev, { _id: tempId, prompt: text, loading: true }]);

    // GỬI REQUEST VỀ BACKEND — đây là lúc "lá thư" được gửi đi!
    const res = await api.post(`/chats/${activeThreadId}`, { prompt: text });
    
    // Nhận "thư trả lời" → thay thế tin nhắn tạm bằng data thật
    setChats((prev) => prev.map((c) => (c._id === tempId ? res.data : c)));
};

Bước 3–5: backend/server.js — Request đi qua hàng loạt "trạm kiểm soát" (middleware)

// Mỗi request đều phải đi qua các middleware theo THỨ TỰ này:
app.use(cors({...}));           // Bước 3: Kiểm tra "anh đến từ đâu?" → cho phép/từ chối
app.use(express.json());        //         Đọc body JSON của request
app.use(session({...}));        // Bước 4: Đọc cookie → khôi phục session cũ
app.use(passport.initialize()); // Bước 5: Khởi tạo Passport
app.use(passport.session());    //         Gọi deserializeUser → tìm user từ DB → gắn vào req.user
Middleware là gì? Tưởng tượng bạn vào công ty: phải qua cổng bảo vệ → quẹt thẻ → đi thang máy → mới đến bàn làm việc. Mỗi "trạm" đó là một middleware. Request cũng phải qua từng trạm trước khi đến được Controller xử lý.

Bước 6: backend/middleware/auth.js — Kiểm tra đăng nhập

export const isAuthenticated = (req, res, next) => {
    if (req.isAuthenticated()) {  // Passport đã tìm được user trong session chưa?
      return next();              // Rồi → cho đi tiếp vào Controller
    }
    return res.status(401).json({ error: 'Please log in' }); // Chưa → chặn lại!
};

Bước 7–10: backend/controllers/chatController.js — "Bếp" xử lý chính

export const createChat = async (req, res) => {
    // Kiểm tra: thread này có thuộc về user đang đăng nhập không?
    const thread = await Thread.findOne({
        _id: req.params.threadId,      // threadId lấy từ URL (/chats/abc123)

    if (!thread) return res.status(404).json({ error: 'Thread not found' });

        htmlContent: [htmlResponse],
        threadId: thread._id,
    });
    
    res.status(201).json(chat);  // Trả kết quả về cho frontend hiển thị
};

4. Google OAuth khác gì đăng ký / đăng nhập thường?

Các bạn chắc đã quen với nút "Tiếp tục với Google" trên các trang web rồi đúng không? Vậy nó khác gì so với tự đăng ký bằng email + password? Mình so sánh cho dễ thấy nhé:

Đăng ký thường

  • User tự nhập email + password
  • Password được mã hoá rồi lưu trong database của mình
  • Khi đăng nhập → so sánh password đã hash
  • Mình phải tự lo bảo mật password
  • User phải nhớ password (hay quên lắm!)

Google OAuth

  • User đăng nhập qua Google
  • Không hề lưu password — Google xác nhận hộ
  • Nhận thông tin (email, tên, ảnh) từ Google
  • Google chịu trách nhiệm bảo mật
  • User chỉ cần 1 click, không nhớ gì hết

Flow: Đăng ký thường

// controllers/authController.js — Hàm đăng ký
export const signup = async (req, res) => {
    const { email, password, firstName, lastName } = req.body;

    // Kiểm tra: email này đã có ai dùng chưa?
    const existingUser = await User.findOne({ email });
    if (existingUser) {
        return res.status(400).json({ message: "Email already registered" });
    }

    // Tạo user mới — password sẽ được TỰ ĐỘNG mã hoá nhờ hook pre('save')
    const newUser = await new User({
        email,
        accountData: { username: email, password, priviledge: "user" },
        identityData: { firstName, lastName, displayName: `${firstName} ${lastName}` },
    }).save();

    const userResponse = newUser.toObject();
    delete userResponse.accountData.password;  // Tuyệt đối KHÔNG gửi password về client
    res.status(201).json(userResponse);
};

Flow: Đăng nhập thường

// controllers/authController.js — Hàm đăng nhập
export const login = (req, res, next) => {
    // Nhờ Passport dùng LocalStrategy để kiểm tra email + password
    passport.authenticate('local', (err, user, info) => {
        if (err) return res.status(500).json({ error: 'Login failed' });
        if (!user) return res.status(401).json({ error: info.message });

        // Đăng nhập thành công → tạo session
        req.logIn(user, (err) => {            // Passport tạo session + gửi cookie cho browser
            if (err) return res.status(500).json({ error: 'Login failed' });
            const userResponse = user.toObject();
            delete userResponse.accountData.password;
            res.json(userResponse);           // Trả dữ liệu user về frontend
        });
    })(req, res, next);  // Gọi ngay hàm authenticate (IIFE pattern)
};
L45passport.authenticate('local') — Passport sẽ gọi LocalStrategy mà mình đã cấu hình ở passport.js. Nó tìm user bằng email, rồi gọi comparePassword để so sánh.
L50req.logIn(user) — Đây là hàm "ma thuật" của Passport. Nó gọi serializeUser để lưu user._id vào session, rồi gửi cookie cho browser. Từ giờ browser mang theo cookie này mỗi lần gọi API.

Flow: Google OAuth — phức tạp hơn nhưng đừng sợ!

Ví dụ thực tế cho dễ hiểu: Google OAuth giống như bạn đến ngân hàng nhưng quên CMND. Bạn nhờ bố mẹ (Google) gọi điện xác nhận: "Đúng rồi, đây là con tôi." Ngân hàng tin tưởng bố mẹ bạn → cho bạn vào. Bạn không cần tạo mật khẩu riêng cho ngân hàng.

Bước 1: Ở đây có một chi tiết quan trọng mà nhiều bạn hay nhầm. Nút "Tiếp tục với Google" dùng thẻ <a href> chứ KHÔNG phải gọi Axios:

<!-- LoginPage.jsx — Dùng thẻ <a> thường, KHÔNG phải api.get -->
<a href="http://localhost:5000/api/auth/google">
    Continue with Google
</a>
L79Tại sao phải dùng <a href>? Vì browser cần chuyển hướng nguyên cả trang sang Google (accounts.google.com). Axios chỉ gửi request "ngầm" bên dưới, không redirect được. Đây là điểm khác biệt rất hay gặp lỗi.

Bước 2: Backend chuyển hướng user đến trang đăng nhập Google:

// controllers/authController.js
export const googleAuth = passport.authenticate('google', {
    scope: ['email', 'profile'],  // "Xin phép" Google cho mình xem email và thông tin cá nhân
});

Bước 5–6: Sau khi user cho phép, Google gọi lại callback URL. GoogleStrategy xử lý:

// config/googleStrategy.js
const createGoogleStrategy = () => {
  return new GoogleStrategy({
      clientID: process.env.GOOGLE_CLIENT_ID,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET,
      callbackURL: 'http://localhost:5000/api/auth/google/callback',
  },
  async (accessToken, refreshToken, profile, done) => {
      const email = profile.emails[0].value;
      
      // Tìm xem email này đã có trong database chưa?
      let user = await User.findOne({ email });
      if (user) return done(null, user);    // Có rồi → đăng nhập luôn, đơn giản!

      // Chưa có → Tạo tài khoản MỚI, nhưng KHÔNG CÓ PASSWORD!
      user = await new User({
          email,
          accountData: { username: email, priviledge: 'user' },
          identityData: {
              firstName: profile.name.givenName || '',
              lastName: profile.name.familyName || '',
              displayName: profile.displayName || '',
              profilePicture: profile.photos[0]?.value || '',  // Lấy luôn ảnh từ Google
          },
      }).save();

      done(null, user);
  });
};
L6clientID + clientSecret — Lấy từ Google Cloud Console. Giống như "thẻ căn cước" của app FinBud khi nói chuyện với Google: "Tôi là FinBud, đây là mã xác minh của tôi."
L9callbackURL — Sau khi user chọn tài khoản Google xong, Google sẽ chuyển hướng về đúng URL này để backend xử lý tiếp.
L11profile — Đây là thông tin Google gửi về: tên, email, ảnh đại diện. Chú ý: mình KHÔNG BAO GIỜ thấy password của họ — Google giữ phần đó.
L14Nếu user đã tồn tại → chỉ cần đăng nhập, KHÔNG tạo mới. Điều này đảm bảo: 1 email = 1 tài khoản, dù đăng nhập bằng Google hay bằng password.
L17Nếu user chưa tồn tại → tạo tài khoản mới không có password. Nên user này chỉ có thể đăng nhập bằng Google mà thôi.

Bước 7: Tạo session rồi chuyển hướng về frontend:

// controllers/authController.js
export const googleCallback = (req, res, next) => {
    passport.authenticate('google', (err, user) => {
      if (err || !user) {
        return res.redirect('http://localhost:5173/login?error=auth_failed');
      }
      req.logIn(user, (err) => {        // Tạo session — GIỐNG HỆT đăng nhập thường!
        if (err) return res.redirect('http://localhost:5173/login?error=auth_failed');
        res.redirect('http://localhost:5173');  // Đưa user về trang chủ frontend
      });
    })(req, res, next);
};
Điểm mấu chốt: Dù đăng nhập bằng Google hay bằng email+password, cuối cùng đều gọi req.logIn() để tạo session. Từ đây trở đi, cả hai cách đều hoạt động y hệt nhau — backend không cần phân biệt nữa.

5. Session hoạt động như thế nào?

Đây là phần nhiều bạn hay mơ hồ nhất. Mình giải thích bằng ví dụ thực tế nhé.

HTTP vốn là vô trạng thái (stateless) — nghĩa là mỗi request đều là một "người lạ". Giống như bạn gọi điện đến tổng đài mà mỗi lần gọi lại gặp nhân viên khác, phải giới thiệu lại từ đầu. Session sinh ra để giải quyết vấn đề đó — nó là cách server "nhớ" bạn là ai giữa các request.

Ví dụ cho dễ hiểu: Session giống như vòng tay vào hội chợ. Lần đầu bạn mua vé (đăng nhập) → nhận vòng tay (cookie). Sau đó mỗi lần đi qua cổng (mỗi request), bạn chỉ cần giơ vòng tay lên → bảo vệ quét mã → biết bạn là ai → cho vào. Hết ngày (cookie hết hạn) → phải mua vé lại (đăng nhập lại).

Bước 1: Cấu hình Session trên server

// backend/server.js
app.use(session({
  secret: process.env.SESSION_SECRET || 'finbud-secret',  // Khoá bí mật để "ký" cookie
  resave: false,             // Không lưu lại session nếu không có thay đổi
  saveUninitialized: false,  // Không tạo session cho request chưa đăng nhập
  cookie: { maxAge: 24 * 60 * 60 * 1000 }  // Cookie hết hạn sau 24 giờ
}));
L34secret — Dùng để ký (sign) cookie, đảm bảo không ai có thể giả mạo. Giống như con dấu của server vậy — chỉ server mới có.
L35resave: false — Tiết kiệm: chỉ cập nhật session khi có gì đó thay đổi.
L37maxAge: 86.400.000ms = 24 giờ — Sau 24 giờ, cookie tự hết hạn, user phải đăng nhập lại. Giống vé tháng xe buýt — hết hạn thì phải mua mới.

Bước 2: Passport Serialize / Deserialize — "Ghi sổ" và "Tra sổ"

// config/passport.js

// KHI ĐĂNG NHẬP: "ghi sổ" — lưu user._id vào session
passport.serializeUser((user, done) => {
    done(null, user.id);  // Chỉ lưu mỗi _id thôi, KHÔNG lưu hết cả object
});

// MỖI REQUEST SAU ĐÓ: "tra sổ" — lấy _id từ session, tìm user trong DB
passport.deserializeUser(async (id, done) => {
    const user = await User.findById(id);  // Truy vấn DB bằng _id
    done(null, user);  // Gắn kết quả vào req.user
});
L35serializeUser — Chạy 1 lần duy nhất khi user đăng nhập. Chỉ lưu user._id vào session (nhẹ, nhanh). Tại sao không lưu cả object? Vì session nằm trong bộ nhớ, lưu ít thì tiết kiệm hơn.
L40deserializeUser — Chạy MỖI REQUEST. Đọc user._id từ session → truy vấn DB lấy đầy đủ thông tin → gắn vào req.user. Nhờ vậy mà Controller luôn biết "ai đang gọi API".

Bước 3: Sử dụng req.user trong Controller

// Ví dụ: goalController.js
export const getGoals = async (req, res) => {
    // req.user đã được Passport gắn sẵn từ deserializeUser
    const goals = await Goal.find({ userId: req.user._id });  // Chỉ lấy goals của user này
    res.json(goals);
};

Tóm tắt toàn bộ flow Session: Đăng nhập → serializeUser lưu user._id vào session → server gửi cookie cho browser → mỗi request sau đó browser gửi kèm cookie → server đọc session → deserializeUser tìm user trong DB → gắn vào req.user → Controller dùng req.user._id để lọc dữ liệu.

6. Tích hợp OpenAI vào dự án ra sao?

Phần này chắc là phần thú vị nhất rồi! FinBud dùng OpenAI GPT-4o-mini để làm chatbot trợ lý tài chính. Và tin vui là: toàn bộ logic tích hợp chỉ nằm trong 1 file duy nhất, tổng cộng 34 dòng code thôi.

Mở file backend/services/aiService.js ra nhé:

import OpenAI from 'openai';

// Singleton pattern: chỉ tạo 1 client duy nhất, dùng đi dùng lại
let openai;
function getClient() {
  if (!openai) {
    openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
  }
  return openai;
}
L1import OpenAI — Đây là SDK chính thức từ OpenAI. Cài bằng lệnh npm install openai.
L3-9Singleton pattern — Chỉ tạo client 1 lần đầu tiên thôi, các lần sau tái sử dụng. Giống như bạn chỉ cần mở cửa 1 lần, sau đó cứ ra vào thoải mái. Tiết kiệm bộ nhớ và thời gian khởi tạo.
L7apiKey — Lấy từ biến môi trường (.env). Đây giống như "mật khẩu" để truy cập OpenAI API. TUYỆT ĐỐI KHÔNG được commit file .env lên git!
export const getAIResponse = async (prompt, chatHistory = []) => {
  // Xây dựng mảng messages theo format OpenAI yêu cầu
  const messages = [
    {
      role: 'system',   // Tin nhắn hệ thống: định nghĩa "tính cách" cho AI
      content: 'You are FinBud, a helpful financial assistant. You help users 
               with stock analysis, financial planning, investment advice, 
               and general financial questions. Be concise and informative.',
    },
  ];

  // Thêm lịch sử chat cũ để AI có "ngữ cảnh"
  for (const chat of chatHistory) {
    messages.push({ role: 'user', content: chat.prompt });
    messages.push({ role: 'assistant', content: chat.response[0] });
  }

  // Thêm câu hỏi mới nhất của user
  messages.push({ role: 'user', content: prompt });

  // Gọi OpenAI API
  const completion = await getClient().chat.completions.create({
    model: 'gpt-4o-mini',   // Model sử dụng (rẻ + nhanh)
    messages,                // Gửi toàn bộ lịch sử + câu hỏi mới
  });

  return completion.choices[0].message.content;  // Lấy nội dung câu trả lời
};
L12messages array — OpenAI API yêu cầu gửi 1 mảng tin nhắn với 3 vai trò: system (định nghĩa bot), user (câu hỏi), assistant (câu trả lời trước đó).
L14-16System message — Gửi 1 lần đầu tiên. Nó dặn dò AI: "Mày là FinBud, trợ lý tài chính, trả lời ngắn gọn nhé." User không nhìn thấy tin nhắn này đâu.
L20-23Lịch sử chat — Đây là mấu chốt để AI "nhớ" được cuộc hội thoại. Thật ra AI không "nhớ" gì cả — mỗi lần gọi API, mình phải gửi LẠI TOÀN BỘ lịch sử. AI đọc lại tất cả rồi mới trả lời câu cuối.
L29gpt-4o-mini — Phiên bản nhỏ, rẻ hơn GPT-4o khoảng 15 lần nhưng vẫn thông minh. Phù hợp cho chatbot.
L33completion.choices[0].message.content — OpenAI có thể trả về nhiều "lựa chọn" (thường là 1). Mình lấy nội dung text từ lựa chọn đầu tiên.
